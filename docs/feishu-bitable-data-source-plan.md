# 小龙虾 AI 自动运营中控台接入飞书多维表格数据源方案

## 1. 背景与目标

当前系统使用本地 JSON 文件作为订单、评论和竞品数据源：

- `orders.json`：订单与销售数据
- `comments.json`：评论与舆情数据
- `competitors.json`：竞品与促销数据

这种方式适合 Demo 阶段快速验证日报生成、飞书推送和首页经营看板能力，但不适合真实客户长期维护。真实门店老板或运营人员通常不会直接登录服务器修改 JSON 文件，也不适合多人协作录入。

飞书多维表格可以作为低成本、低开发量的数据后台。客户可以直接在飞书里录入和维护订单、评论、竞品信息，系统定时读取表格数据后生成经营日报。

本方案目标：

- 让客户在飞书多维表格里维护经营数据
- 系统读取飞书表格数据并转换为当前日报生成链路可用的数据结构
- 保留现有 JSON 数据源作为默认和回退方案
- 在不引入数据库的前提下，提高真实客户试用时的数据维护便利性

## 2. 当前 JSON 数据源的问题

当前 JSON 数据源主要问题如下：

- 客户无法直接维护服务器上的 JSON 文件，必须依赖开发或运维人员操作。
- 数据录入不方便，尤其是订单、评论、竞品信息每天都可能变化。
- 不利于多人协作，例如前厅、运营、店长无法同时维护不同类型的数据。
- 不利于后续扩展真实业务数据，例如多门店、多平台、多日期范围查询。
- JSON 文件缺少天然的权限、编辑记录和表格视图，客户理解和使用成本较高。

因此，JSON 更适合作为 Demo 默认数据源和兜底数据源；真实客户试用阶段，更适合引入飞书多维表格作为轻量数据后台。

## 3. 推荐表结构

建议在同一个飞书多维表格应用中创建 3 张表：订单表、评论表、竞品表。

### 订单表

| 字段 | 说明 |
| --- | --- |
| 日期 | 数据所属日期，例如 `2026-05-16` |
| 时段 | 午市、晚市、夜宵等 |
| 订单数 | 当前时段订单数量 |
| 销售额 | 当前时段销售金额 |
| 客单价 | 当前时段平均客单价 |
| 爆款产品 | 当前时段或当天主推、热卖产品 |
| 平台 | 美团、饿了么、堂食、私域等 |
| 备注 | 异常情况、活动说明或人工补充 |

### 评论表

| 字段 | 说明 |
| --- | --- |
| 日期 | 评论日期 |
| 平台 | 美团、饿了么、抖音、本地生活等 |
| 评分 | 用户评分 |
| 评论内容 | 用户原始评论 |
| 风险关键词 | 慢、不够入味、少、服务差等 |
| 是否差评 | 是 / 否 |
| 处理状态 | 未处理、已回复、已同步后厨、已解决等 |
| 备注 | 处理说明、补偿记录或复查结果 |

### 竞品表

| 字段 | 说明 |
| --- | --- |
| 日期 | 观察日期 |
| 竞品门店 | 周边竞品门店名称 |
| 平台 | 美团、饿了么、抖音、本地生活等 |
| 促销动作 | 满减、折扣、套餐、夜宵加购等 |
| 热卖品 | 竞品主推或销量较高的产品 |
| 备注 | 人工观察补充 |

## 4. 字段映射关系

飞书多维表格读取后，需要转换为当前系统可消费的数据结构。第一版建议只做字段级映射，不改变现有日报生成链路。

### 订单表到 `orders`

| 飞书字段 | 当前系统字段建议 | 说明 |
| --- | --- | --- |
| 日期 | `date` | 字符串日期 |
| 时段 | `period` 或 `time_slot` | 午市、晚市、夜宵 |
| 订单数 | `order_count` | 数值 |
| 销售额 | `revenue` | 数值 |
| 客单价 | `average_order_value` | 数值，可由表格提供，也可后续由系统计算 |
| 爆款产品 | `top_product` | 字符串 |
| 平台 | `platform` | 字符串 |
| 备注 | `note` | 字符串，可选 |

### 评论表到 `comments`

| 飞书字段 | 当前系统字段建议 | 说明 |
| --- | --- | --- |
| 日期 | `date` | 字符串日期 |
| 平台 | `platform` | 字符串 |
| 评分 | `rating` | 数值 |
| 评论内容 | `content` | 用户评论原文 |
| 风险关键词 | `risk_keywords` | 字符串或字符串列表 |
| 是否差评 | `is_negative` | 布尔值，`是` 转为 `true` |
| 处理状态 | `status` | 未处理、已回复等 |
| 备注 | `note` | 字符串，可选 |

### 竞品表到 `competitors`

| 飞书字段 | 当前系统字段建议 | 说明 |
| --- | --- | --- |
| 日期 | `date` | 字符串日期 |
| 竞品门店 | `store_name` | 竞品门店名称 |
| 平台 | `platform` | 字符串 |
| 促销动作 | `promotion` | 满减、套餐、折扣等 |
| 热卖品 | `hot_product` | 字符串 |
| 备注 | `note` | 字符串，可选 |

映射原则：

- 飞书字段名面向客户，尽量中文、直观。
- 系统内部字段保持稳定，避免影响现有 Agent 和日报生成逻辑。
- 字段缺失或类型异常时，先记录 warning，再尝试跳过异常记录或使用默认值。
- 不要求客户理解 JSON，只需要维护表格。

## 5. 环境变量设计

建议预留以下环境变量：

```bash
DATA_SOURCE=json
FEISHU_BITABLE_APP_TOKEN=
FEISHU_ORDERS_TABLE_ID=
FEISHU_COMMENTS_TABLE_ID=
FEISHU_COMPETITORS_TABLE_ID=
```

字段说明：

- `DATA_SOURCE=json`：默认继续使用本地 JSON 数据源。
- `DATA_SOURCE=feishu_bitable`：启用飞书多维表格数据源。
- `FEISHU_BITABLE_APP_TOKEN`：飞书多维表格应用 token。
- `FEISHU_ORDERS_TABLE_ID`：订单表 table id。
- `FEISHU_COMMENTS_TABLE_ID`：评论表 table id。
- `FEISHU_COMPETITORS_TABLE_ID`：竞品表 table id。

后续如需要调用飞书开放平台接口，还需要复用或新增飞书应用凭证，例如 `FEISHU_APP_ID`、`FEISHU_APP_SECRET`。这些密钥只能从 `.env` 或系统环境变量读取，不应写入代码或日志。

## 6. 分阶段接入方案

### 第一阶段：只写方案文档，不改代码

明确飞书多维表格作为数据源的表结构、字段映射、环境变量和接入边界。

本阶段不修改日报生成链路，不修改首页，不修改飞书事件订阅，也不改变现有 JSON 数据源。

### 第二阶段：新增 debug 接口，验证能否读取飞书多维表格

新增一个仅用于调试的接口，例如：

```text
GET /api/debug/feishu-bitable/orders
```

目标是验证：

- 飞书应用权限是否正确
- `app_token` 和 `table_id` 是否正确
- 表格字段是否能读取
- 返回结构是否符合预期

该阶段只验证读取能力，不接入日报生成链路。

### 第三阶段：增加 `DATA_SOURCE` 开关，支持 `json` / `feishu_bitable` 切换

在数据读取层增加数据源选择逻辑：

- `DATA_SOURCE=json`：继续读取本地 `orders.json`、`comments.json`、`competitors.json`
- `DATA_SOURCE=feishu_bitable`：读取飞书多维表格并转换为系统内部结构

建议优先在 `json_store` 或新增的数据源 service 层处理，不要让 Agent 直接感知飞书接口。

### 第四阶段：日报生成链路正式支持飞书多维表格

当 `DATA_SOURCE=feishu_bitable` 时：

- 定时任务 `/api/reports/daily` 使用飞书表格数据生成日报
- 飞书群指令“生成今日日报”使用飞书表格数据生成日报
- 首页继续只读展示最近一次生成结果

接口返回结构、飞书推送逻辑、历史留存逻辑保持不变。

### 第五阶段：完善异常处理、日志和回退机制

完善以下能力：

- 飞书读取失败时记录清晰 warning
- 不打印 `app_token`、`table_id`、密钥或完整响应
- API 超时、权限失败、限流时可回退到 JSON
- 字段缺失、字段类型异常时跳过异常记录或降级处理
- 日志能区分当前使用的数据源：`json` 或 `feishu_bitable`

## 7. 回退方案

默认继续使用：

```bash
DATA_SOURCE=json
```

回退原则：

- 飞书多维表格读取失败时，不影响原有 JSON 流程。
- 不让飞书数据源失败拖垮日报生成。
- `DATA_SOURCE=json` 始终作为稳定兜底方案。
- 如果 `DATA_SOURCE=feishu_bitable` 但配置缺失，应记录 warning，并回退到 JSON 或直接给出明确错误，具体策略在第二阶段验证后确定。
- 飞书 API 异常、超时、限流或字段解析失败时，不应导致首页或飞书事件接口崩溃。

建议第一版正式接入时采用“读取飞书失败则回退 JSON”的保守策略，保证客户仍能收到日报。

## 8. 风险点

### 飞书应用权限配置复杂

多维表格读取通常需要配置应用权限、授权范围和表格访问权限。权限缺失会导致接口返回失败。

### `app_token`、`table_id`、字段配置容易填错

飞书多维表格涉及应用 token、表 id、字段名等配置。任何一个填错都会导致读取不到数据或字段映射失败。

### 字段类型可能不一致

客户可能把数值字段填成文本，把日期填成非标准格式，或把“是否差评”填成多种表达。系统需要做基础清洗和容错。

### API 限流

飞书接口可能存在限流。日报生成虽然频率不高，但如果后续增加多门店、多表读取或历史同步，需要注意请求频率。

### 客户录入数据不规范

客户可能漏填日期、订单数、评论内容，或使用不同的促销描述。第一版需要接受一定程度的不完整数据，并在日报里避免过度解读。

## 9. 后续可扩展方向

- 接入真实门店订单系统，减少人工录入订单数据。
- 支持多门店，在表格中增加门店字段，日报按门店生成或汇总。
- 支持在表格里直接标记差评处理状态，日报优先提醒未处理差评。
- 支持飞书日报生成记录回写，把生成时间、摘要、处理建议写回多维表格。
- 支持老板在飞书里查看历史日报，例如通过“查看昨天日报”“查看本周日报”等指令读取历史记录。
- 支持按平台拆分经营表现，例如美团、饿了么、抖音分别统计订单、评论和竞品动作。
- 支持数据质量提醒，例如提示“今日评论表缺少处理状态”或“订单表缺少夜宵数据”。
